## - [x] 1. 완성된 코드 — 완성도

사이트 들어가보니까 방명록이 잘 동작하고 있어요! 글 남기기, 목록 불러오기, 삭제까지 기본 CRUD 중 CRD가 구현되어 있네요.

**잘 된 부분:**
- 3계층 구조(GitHub Pages → Render → Supabase)가 깔끔하게 나뉘어 있어서 이해하기 쉬웠어요
- 프론트엔드에 비밀 키가 하나도 없는 구조가 인상적이었어요. 저는 Supabase를 프론트에서 직접 연결했는데, 이 프로젝트는 백엔드를 따로 두니까 아예 키 노출 걱정이 없더라고요
- 코드 주석이 `[1]~[6]` 번호 순서로 되어 있어서 처음 보는 사람도 흐름을 따라가기 좋았어요
- README도 배포 순서까지 상세하게 적혀 있어서 따라 하기 쉬울 것 같아요

**아쉬운 부분:**
- 삭제를 누구나 할 수 있어서, 다른 사람이 쓴 글도 지울 수 있는 게 좀 걸렸어요. 물론 "공개 방명록"이라 의도된 거 같긴 한데, 실서비스면 좀 위험할 수도 있겠다 싶었어요
- 수정(Update) 기능은 없어서 글을 잘못 쓰면 지우고 다시 써야 하는 점이 살짝 불편할 수 있겠어요
- UI가 정말 심플한데, 그게 오히려 깔끔하다는 느낌도 있었어요

## - [x] 2. 문제 해결 — 보안 체크리스트

### ① 비밀값을 코드에 직접 안 적고 환경변수로 뺐는가?

**네, 잘 되어 있어요!** `server.js`에서 `process.env`로 `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`를 읽어오고, `.env.example`도 잘 만들어뒀어요. `index.html`에는 API 주소(`API_BASE_URL`)만 있고 키가 아예 없어서, 제 프로젝트보다 이 부분은 더 안전한 구조라고 느꼈어요.

### ② 가격·수량·권한 같은 검증을 백엔드에서 하는가?

**네, 하고 있어요.** `server.js:47-49`에서 글 내용이 비어 있는지, 500자 넘는지 서버에서 검증하고 있어요. 프론트의 `maxlength="500"`만 믿지 않고 서버에서도 체크하는 게 좋았어요. DB에도 `CHECK(char_length(content) <= 500)` 제약이 있어서 3중 검증이네요. 이건 저도 배워야 할 부분이었어요.

### ③ Supabase를 쓴다면 RLS를 켜고 정책을 넣었는가?

**RLS를 켜고, 정책은 일부러 안 넣은 구조예요.** `db.sql`에 `enable row level security`가 있고, 정책을 만들지 않으면 브라우저에서 직접 접근이 차단된다고 설명이 되어 있어요. 이 프로젝트는 백엔드에서 `service_role` 키로 접근하니까 RLS를 우회하고, 브라우저는 어차피 DB에 직접 접근 안 하니까 이게 맞는 것 같아요. 처음에 "정책이 없네?" 했다가 구조를 이해하고 나니 납득이 됐어요.

### ④ 로그인이 필요한 데이터에 권한 확인(인가)이 들어가는가?

**로그인 기능 자체가 없는 공개 방명록이에요.** 그래서 인가(authorization) 체크가 따로 없어요. 누구나 글 쓰고 누구나 삭제할 수 있는 구조인데, README에도 "로그인 없는 공개 방명록"이라고 명시해놔서 의도된 설계라는 게 느껴졌어요. 다만 악의적으로 남의 글을 삭제하는 건 막을 수 없으니, 나중에 로그인을 붙이면 좋겠다는 생각은 들었어요.

### ⑤ 사용하는 무료 티어의 한도와 요금 알림을 아는가?

README에 "무료 Supabase/Render는 방치 시 잠들 수 있다"고 적혀 있고, Render 콜드 스타트도 언급했어요. 한도를 인지하고 있는 것 같아요. 다만 구체적인 수치(Supabase 500MB, Render 750시간 등)까지는 안 적혀 있어서, 코더한테 물어보면 좋겠어요.

## - [x] 3. 프로젝트에 대한 회고가 상세히 기록되어 있나요? (회고, 정리)

- [x] **배운 점**: 직접 프론트엔드와 백엔드 DB 서버를 구분하면서 구현에 필요한 것들을 이해하고 구분하기 시작했습니다.

- [x] **아쉬운 점**: 프론트엔드에 소셜 로그인 기능을 구현하지 못한 것이 아쉽습니다.

- [x] **느낀 점**: AI를 사용한다면, 웹&앱 등을 만들 수 있겠지만, 예로 웹의 기술적 전반적인 지식이 부족하다면 결과물 완성도에 큰 차이가 생길 것 같습니다.

- [x] **어려웠던 점**: AI가 기술적 설명을 할 때, 그 설명에서 이해 못하는 단어들이 많아서 다시 재차 물어보고 찾아보고 반복하다보니, 원래 하고자 하던 것이 무엇이었는지조차 잊어버리는 상태가 왔었습니다.
